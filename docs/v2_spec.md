# qr-inventory-system Ver2.0 仕様書

## Ver2.0の目的

Ver2.0では、Ver1.0で実装したデスクトップ版の在庫管理機能を残したまま、スマートフォンやPCブラウザから利用できるWebアプリ版を追加開発する。
現場担当者がQRコードを読み取り、端末に専用アプリをインストールせずに入庫・出庫・在庫確認を行えることを主目的とする。

主な目的は以下とする。

- 既存のCUI版・GUI版の資産を活かし、Webブラウザから操作できる画面を追加する。
- スマートフォンでのQRコード読み取りと現場入力をしやすくする。
- PCブラウザでは品目一覧、履歴確認、マスタ管理、バックアップなどの管理作業を行いやすくする。
- 小規模な社内利用から、将来的な外部公開・販売を見据えた構成へ段階的に移行する。

## Ver1.0との差分

Ver1.0はローカルPC上で動作するデスクトップ版を中心とする。
Ver2.0は既存機能を廃止せず、同じ在庫データをWebアプリから扱えるようにする。

| 項目 | Ver1.0 | Ver2.0 |
| --- | --- | --- |
| 利用形態 | CUI / デスクトップGUI | Webブラウザ |
| 主な端末 | 管理PC | PC / スマートフォン / タブレット |
| 起動方法 | `python src/main.py` / `python src/gui_main.py` | `uvicorn` でWebサーバー起動 |
| QRコード利用 | PNG生成・印刷用HTML生成 | スマートフォンカメラで読み取り、Web画面へ反映 |
| 認証 | GUI管理者パスワード中心 | ログイン、ロール、セッション管理を導入予定 |
| 公開範囲 | ローカルPC | ローカルLAN、トンネル経由確認、VPS公開へ段階移行 |
| DB | SQLite | 当面SQLite、将来PostgreSQLへ移行可能に設計 |

## Webアプリ化方針

Webアプリ版は、既存の業務ロジックとDB構造をできるだけ活かしながら、HTTP経由で操作できるUIを追加する。
初期段階ではサーバーサイドレンダリングを基本とし、FastAPI、Jinja2、HTML/CSS/JavaScriptで画面を構築する。

方針は以下とする。

- 既存のデスクトップ版は削除せず、Ver1.0互換の操作手段として維持する。
- 在庫操作、品目検索、履歴表示、最低在庫アラートなどの共通ロジックは再利用しやすい形に整理する。
- Web固有の処理（ルーティング、テンプレート、フォーム検証、認証）は既存ロジックと分離する。
- 画面遷移は単純で、現場作業者が迷わない構成にする。
- 初期リリースでは複雑なSPA化を避け、保守しやすい構成を優先する。

## スマートフォン対応方針

スマートフォン対応では、現場で片手操作しやすい画面とQRコード読み取りを重視する。

- レスポンシブデザインを採用し、幅の狭い画面でも主要操作を行えるようにする。
- 入庫・出庫・品目検索はスマートフォンの操作を最優先に設計する。
- ボタン、入力欄、リンクはタップしやすいサイズにする。
- QRコード読み取りはブラウザのカメラAPI利用を検討する。
- カメラAPIが使えない端末向けに、品目IDやQRコード文字列の手入力も残す。
- 通信が不安定な現場を想定し、送信後の結果表示、二重送信防止、エラー表示を明確にする。

## 違うWi-Fiから利用する方法

ローカルPCで起動したWebアプリは、通常は同じLAN内からのアクセスに限定される。
違うWi-Fi、社外、モバイル回線から利用する場合は、以下のいずれかの方法を検討する。

1. **ngrok / Cloudflare Tunnel で一時的に公開する**
   - 開発・デモ・外部確認向け。
   - ローカルPCで起動したWebアプリへインターネット経由でアクセスできる。
   - URLが外部に公開されるため、必ず認証を有効化する。

2. **VPSへデプロイする**
   - 本番運用・継続利用向け。
   - 固定ドメイン、HTTPS、バックアップ、監視、OSアップデートを含めて運用する。
   - SQLiteからPostgreSQLへの移行も検討する。

3. **VPNを利用する**
   - 社内システムとして閉域利用したい場合の選択肢。
   - WireGuard、Tailscale、Zero Trust系サービスなどを検討する。

## Step 1: ローカルWeb化

最初の段階では、開発PCまたは社内PC上でWebサーバーを起動し、同じPCまたは同じWi-Fi内の端末からアクセスする。

- FastAPIアプリを追加する。
- Jinja2テンプレートで画面を作成する。
- Uvicornでローカル起動する。
- SQLiteの既存DBを利用する。
- 品目検索、品目一覧、入庫、出庫、最低在庫アラートを優先してWeb化する。
- 管理機能は通常操作が安定してから段階的に追加する。

起動イメージ:

```bash
uvicorn src.web_app:app --host 0.0.0.0 --port 8000
```

同じWi-Fi内のスマートフォンからは、PCのローカルIPアドレスを指定してアクセスする。

```text
http://<PCのローカルIPアドレス>:8000
```

## Step 2: ngrok / Cloudflare Tunnel による外部確認

ローカルWeb化後、違うWi-Fiやモバイル回線から確認するために、ngrokまたはCloudflare Tunnelを利用する。
この段階は本番公開ではなく、動作確認、顧客デモ、スマートフォン実機確認を目的とする。

- 外部公開URLを発行し、スマートフォンからアクセス確認する。
- ログイン機能または最低限のBasic認証を必ず有効化する。
- 公開URLを不用意に共有しない。
- デモ用データと本番データを分ける。
- トンネルの起動・停止手順をREADMEまたは運用メモに記録する。

利用イメージ:

```bash
uvicorn src.web_app:app --host 127.0.0.1 --port 8000
ngrok http 8000
```

Cloudflare Tunnelを利用する場合は、Cloudflareアカウント、ドメイン、トンネル設定、アクセス制御を確認する。

## Step 3: VPS公開

本番運用や販売先での継続利用を想定する段階では、VPSへデプロイする。

- Linux VPS上でアプリを稼働する。
- Uvicorn単体ではなく、必要に応じてGunicornやsystemdと組み合わせる。
- Nginxなどのリバースプロキシを前段に置く。
- HTTPSを必須とする。
- 独自ドメインを設定する。
- DBバックアップ、ログローテーション、監視、障害復旧手順を整備する。
- 管理者アカウント、一般ユーザーアカウント、初期パスワード変更を必須にする。

## 販売前提のオープンソース方針

本プロジェクトは販売や受託導入を見据えつつ、オープンソースとして公開できる形を維持する。
公開リポジトリには汎用機能を置き、顧客固有の設定、秘密情報、個別カスタマイズは分離する。

方針は以下とする。

- ソースコードは公開可能な内容に限定する。
- APIキー、パスワード、顧客データ、実在庫データをコミットしない。
- 導入支援、設定代行、保守、追加開発を販売対象にできる構成にする。
- 汎用版と顧客別カスタマイズを分け、顧客固有コードの混入を避ける。
- README、仕様書、セットアップ手順を整備し、第三者が検証できる状態を保つ。

## ライセンス方針

現時点ではMIT Licenseを前提とする。
MIT Licenseは商用利用、改変、再配布がしやすく、販売前提の小規模業務アプリと相性がよい。

- リポジトリのライセンス表記と実際の利用ライブラリのライセンスを定期的に確認する。
- 依存ライブラリを追加する場合は、ライセンスを確認してから採用する。
- 顧客へ提供する場合は、ライセンス表示、著作権表示、第三者ライブラリのライセンス表記を同梱する。
- 独自の有償サポート、導入支援、ホスティングサービスはMIT Licenseと矛盾しない範囲で提供する。

## GPL / AGPL 系ライブラリを避ける方針

販売、SaaS提供、顧客別カスタマイズを想定するため、GPL / AGPL系ライブラリの採用は原則避ける。
特にAGPLはネットワーク越しの利用でもソースコード公開義務が発生し得るため、Webアプリ版では注意する。

- 新規ライブラリ採用時にライセンスを確認する。
- MIT、BSD、Apache-2.0など、商用利用しやすいライセンスを優先する。
- GPL / AGPL / LGPLのライブラリが必要な場合は、代替ライブラリを先に検討する。
- どうしても採用が必要な場合は、影響範囲、配布形態、ソース公開義務を確認してから判断する。

## 推奨技術構成

Ver2.0の初期構成は、既存のPython資産を活かし、学習コストと保守性のバランスを重視する。

### FastAPI

- Webアプリのルーティング、フォーム処理、API化に利用する。
- 将来的にJSON APIを追加しやすい。
- 型ヒントを活用して保守性を高める。

### Jinja2

- サーバーサイドHTMLテンプレートとして利用する。
- 初期段階ではSPAよりもJinja2で画面を作る。
- 画面共通レイアウト、通常メニュー、管理者メニューをテンプレート化する。

### Uvicorn

- FastAPIのASGIサーバーとして利用する。
- 開発時はUvicorn単体で起動する。
- 本番ではsystemd、Nginx、必要に応じてGunicornなどと組み合わせる。

### SQLite

- 初期段階では既存のSQLite DBを利用する。
- 小規模運用、単一拠点、少人数利用ではSQLiteで開始する。
- 同時アクセスが増える場合はPostgreSQL移行を検討する。

### HTML/CSS/JavaScript

- HTML/CSSは軽量で保守しやすい構成にする。
- JavaScriptはQRコード読み取り、二重送信防止、簡易的な画面補助に限定して始める。
- フレームワーク導入は必要性が明確になってから判断する。

## 将来のPostgreSQL移行方針

将来的に同時利用者数が増える、複数拠点で使う、VPS上で本番運用する場合はPostgreSQLへの移行を検討する。

- DBアクセス層を整理し、SQLite固有処理を局所化する。
- SQLの方言差に注意する。
- マイグレーション手順を用意する。
- SQLiteからPostgreSQLへデータ移行するスクリプトを準備する。
- バックアップ方式をSQLiteのファイルコピーから、PostgreSQLの`pg_dump`等へ切り替える。
- 本番移行前にテストデータで移行検証を行う。

## 認証・権限管理方針

Webアプリ版では、デスクトップ版よりも認証・権限管理を重視する。
特に外部公開、トンネル公開、VPS公開ではログイン必須とする。

- ユーザーIDとパスワードによるログイン機能を導入する。
- パスワードは平文保存せず、ハッシュ化して保存する。
- セッション管理を行い、ログアウト機能を提供する。
- 管理者ユーザーと通常ユーザーを分ける。
- 危険操作は管理者権限を必須にする。
- 初期管理者パスワードは初回起動後に変更必須とする。
- CSRF対策、セッションCookieのSecure/HttpOnly/SameSite属性を検討する。

## 管理者メニュー方針

管理者メニューには、データ変更リスクが高い操作やシステム管理操作を集約する。

管理者メニューに含める機能候補:

- 品目登録
- 品目編集
- 品目削除
- 棚卸修正
- CSV品目マスタ取込
- CSV取込プレビュー
- QRコード生成（単品）
- QRコード生成（全件）
- QRコード印刷用HTML生成
- DBバックアップ
- DB復旧
- ユーザー管理
- システム設定

方針:

- 管理者権限がないユーザーには表示しない。
- 削除、復旧、棚卸修正、CSV取込などは確認画面を挟む。
- 危険操作の直前に自動バックアップを作成する。
- 操作履歴を残せるようにする。

## 通常ユーザーメニュー方針

通常ユーザーメニューは、現場作業者が日常的に使う機能に限定する。

通常ユーザーメニューに含める機能候補:

- 品目検索
- 品目一覧
- 入庫
- 出庫
- 入出庫履歴表示
- 最低在庫アラート
- QRコード読み取りによる品目検索

方針:

- 危険操作やマスタ編集を通常ユーザーに表示しない。
- 入庫・出庫画面はスマートフォンでも素早く操作できるようにする。
- 在庫不足、入力エラー、登録なし品目を分かりやすく表示する。
- 操作完了後に現在庫を明確に表示する。

## バックアップ方針

在庫管理システムではデータ消失や誤操作への備えが重要である。
Ver2.0でも、Ver1.0のバックアップ方針を引き継ぎつつ、Web運用に合わせて強化する。

- SQLite運用では、`data/inventory.db`を定期的にバックアップする。
- 手動バックアップ機能を管理者メニューに用意する。
- 棚卸修正、CSV取込、DB復旧などの危険操作前には自動バックアップを作成する。
- VPS運用では、サーバー外部へのバックアップ保管を検討する。
- バックアップから復旧する手順を文書化し、定期的に復旧テストを行う。
- バックアップファイルには在庫データが含まれるため、アクセス権限と保管場所に注意する。

## セキュリティ注意点

Webアプリ化により、デスクトップ版よりもセキュリティ上の注意点が増える。
特に外部公開時は以下を必須事項として扱う。

- 本番公開ではHTTPSを必須とする。
- 管理者パスワードや秘密情報をソースコードへ直接書かない。
- `.env`や環境変数で設定を管理し、秘密情報をGit管理しない。
- SQLインジェクションを避けるため、SQLはプレースホルダを使う。
- CSRF、XSS、セッション固定、クリックジャッキングへの対策を検討する。
- ファイルアップロードやCSV取込では拡張子、サイズ、文字コード、内容検証を行う。
- 外部公開URLを利用する場合は、認証なしで公開しない。
- 初期パスワードを本番環境で使い続けない。
- ログにパスワードや個人情報を出力しない。
- OS、Python、依存ライブラリを定期的に更新する。
